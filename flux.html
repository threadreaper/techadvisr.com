Welcome to Flux!

Flux is an open source UX/UI design developed and maintained by Techadvisr for use across mixed media brand materials. For now, it includes a .css and a .js file for applying the UX design to website and email materials, along with this document detailing the UX theory and usage.

Why re-invent the wheel? Why not just use Material Design?

When Google introduced its Material Design UX standard, it was quickly adopted by designers everywhere, and is available as itself or in some derivative form through every CMS system commonly in use today. The result of this has been a trend towards a more standardized user experience across both Internet websites and device-level applications. This has the benefit of familiarity for users when they encounter new content. UI components like navigation icons, menu icons, etc are familiar to the user and don't need to be explained. This is objectively a good thing. However, there are several caveats to the wide-spread usage of Material Design and highly-derivative UX standards.

The Internet is boring

In the early days of the World Wide Web, there were no accepted or even suggested standards of UX/UI design. Every website or application one encountered felt unique, and designers had maximum artistic freedom. A professional web designer was forced to think about UI design and the overall UX, which meant that a nice-looking, user-friendly website was much more difficult and expensive to create then than it is now. While Google must be credited for the savings of time, money and frustration achieved by applying one open-source UX design to literally everything, applications and web sites feel less unique because of it. Everything looks like it belongs to Google.

Web Design is for Dummies

Modern CMS packages such as Wordpress and Joomla, along with the widespread usage of Material Design have created a narrow expectation of what a "good" website should like, and anything that doesn't adhere to the current trends is not recognized for its uniqueness, but rather criticized for poor design practices. In this context, anyone can do web design, as there is no longer any aspect of actual design involved. One simply generates a Material Design theme, applies it to whatever Wordpress template is currently trending and inserts a bunch of cookie-cutter content. As a result, many of those marketing themselves as professional web designers today don't have the slightest clue how to dig into html, css or javascript code to fix, modify or customize anything beyond what is available in a template. They avoid revealing the fact that they have no idea what they're doing by countering any outside-the-box request from the client with, "That would be a bad idea, because... SEO."

Speaking of SEO...

Google has not been the best search engine on the Internet in quite awhile. For that, see duckduckgo.com. Yet, due to several years of holding that position, the billions of dollars it generated for them, and a highly successful marketing strategy, it remains the most widely utilized. Thus, designers are forced to optimize for Google's search routine if they hope to generate any noteworthy amount of traffic. Google's (closed-source) algorithm is known to favor websites that follow accessibility and ease-of-use standards that are addressed by its Material Design UX, and thus employing Material Design is seen as a "magic bullet" that will land a site higher on Google's index.

Dispelling the Myths

It is possible to rank well on Google without building a cookie-cutter Wordpress website and using Material design. In fact, a static website unburdened by the overhead required for a CMS to generate pages will load and perform faster, which is known to lead to better ranking in search results. Designing for accessibility and ease of use is important, but it is possible to both achieve this, and pass Google's accessibility tests without producing a website that looks like it was copy/pasted from every other website on the Internet. What follows is a set of UX design standards that address all the accessibility requirements known to affect Google ranking while maintaining the familiarity benefits of Material Design, but discarding its flat, bland, and generally boring aesthetics.

Is it available for Wordpress?

Currently, no. It is open-source, under the MIT license which means free as in free beer. Feel free to convert it for Wordpress yourself if that's what you really want to do. The only thing we ask is that if you use Flux for your website project, you provide credit and a link to Techadvisr.

Typography

Fonts are designed to be responsive and scale to sane values, regardless of viewport size. The initial settings somewhat follow Google's Material Design standards, but as Material Design templates don't seem to have any scaling built in, these should be more universally useful and provide a better cross-platform UX. Two fonts are used and can be set by changing the --font-family-1 and --font-family-2 variables under the :root selector in the CSS. Headlines 1-6 are styled onto the <h1-6> tags. The other settings are classes, except "Body" and "Body 2" which represent the default font styless applied to <body> and <p>, respectively.

Scrollbars

Scrollbars have been themed for browsers that support it. Better cross-browser support is possible, and should be supported in the future.

Color Scheme

The color scheme in use is quite simple and minimalist. The UX is really designed to work around one major color. The easiest implementation is to select your primary color, along with a lightened and darkened shade of the same tone, and set the --primary, --primary-light and --primary-dark variables in the :root selector of the CSS. Everything should work just fine. If you need to use more than one major tone, you can try setting another color up as --secondary, --secondary-light and --secondary-dark, but as these are used for various drop shadows and borders throughout the UX, your mileage may vary. If a one-color simplistic color scheme does not suit your needs, this UX/UI probably isn't a good fit for you.

Wrapper Classes

Three flex-box content wrapper classes are defined for you. .wrapper, .wrapper-offset-secondary, and .wrapper-offset-primary. They apply a neutral, secondary-light and primary-light background, respectively. You should always use these content wrappers, as they have rules defined to make necessary adjustments to element classes, in order to keep the UX consistent across all background colors.

Interactive UI components

Interactive pieces of the UI are designed to be "chunky" enough for ease of use on mobile devices at a scale that is still usable for desktop and larger viewports. This makes implementing a "mobile-first" design easier, keeps the code lightweight, and improves performance. Hover animation are applied to most components on any desktop-size or larger viewport. The hover animations are loaded through media queries in order to prevent mobile devices from loading code they won't utilize. The goal is 100% code coverage capability, meaning no unnecessary code is ever loaded into memory. We'll take the liberty of coining the term, "performance-first UX design."

Iconography

The icons can be generated in the kit's primary, secondary and neutral colors by applying the classes .filter-primary, .filter-secondary and .filter-neutral, respectively. These classes aren't in the CSS, they are dynamically handled by JavaScript, which reads in the customized colors from CSS variables and calculates a filter to apply to the .svg icon. Again, it might seem as though we're re-creating the wheel when fontawesome is a thing, but this iconset has had its code thoroughly optimized for maximum performance and perhaps most importantly, all the icons have the same 1:1 aspect ratio, so they will line up neatly when trying to put them in a row, unlike fontawesome and other such packages. Mouseover effects are handled automatically by JavaScript. Classes have been provided to scale the icons to various sizes, as detailed below.

Alternatively, a modal popup can be triggered when a user scrolls the document a certain percentage. This allows you to present modal popups only to users who seem to be engaged in your content. To trigger your modal in this way, add the following line to the <head> of your document: <style>:root{--modal-trigger: 50;}</style> Where the value of the --modal-trigger variable is set to the scroll percentage at which you want to trigger the modal popup.

And of course, we can also launch an "exit-intent" modal popup when we detect that our user is attempting to leave our page. For this functionality, add this line to the <head> of your html document: <style>:root{--exit-intent: 1;}</style> TODO: Exit intent popups are not working in Firefox or on mobile devices.

The latter 2 methods will check for the existence of a sessionStorage variable indicating the modal popup has been seen before displaying it. If the variable is not set, it will be set before the modal popup is displayed, and any further popup triggers will be ignored for the duration of the session. If you're using the modal for something like adding users to an email list, it is a good idea to provide an obvious route to this functionality elsewhere within your document, in case the user inadvertently closes the dialog without meaning to so you don't lose the conversion.
